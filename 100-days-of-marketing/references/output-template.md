## Review & Adjust

At the end of each phase, answer these questions honestly:

1. **What worked?** → Double down on these
2. **What didn't?** → Cut or change approach
3. **Does this still serve my community?** → If no, realign
4. **Am I spending time or money?** → Favor time in early phases
5. **Is this the simplest approach?** → If not, simplify
6. **Are my customers asking for this?** → If not, reconsider
7. **Am I still excited about this?** → Burnout kills more businesses than competition
8. **What one thing would I change?** → Change it in the next phase

## Section Guidelines

### Daily Tasks
Each daily task must be:
- **Specific**: "Write a 500-word blog post comparing {app} to {competitor} for {use case}" — not "create content"
- **Actionable**: Something the user can sit down and do in one session
- **Time-boxed**: Include realistic time estimate in minutes
- **Connected**: Reference which deliverable from a previous day it builds on, if applicable

### Copy Examples
When the plan includes example copy (headlines, tweets, emails, CTAs):
- Write them in the user's tone from tone-of-voice.md
- Follow the copy rules from SKILL.md Step 4
- Provide 2-3 alternatives for key headlines and CTAs
- Use actual app name, features, and audience — no generic placeholders

### KPI Targets
- Base targets on the user's current stage and time budget
- Pre-launch targets are lower; scaling targets are higher
- Always include how to measure (specific tool or method)
- Be realistic for a solo indie developer
